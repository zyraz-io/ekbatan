# Releasing Ekbatan

This document covers the release pipeline: one-time setup, the per-release workflow, and the safety nets in between. The publishing surface is two targets:

- **15 jars to Maven Central** under groupId `io.github.zyraz-io` (via Sonatype Central Portal, stage-and-confirm). The Java packages in the source tree are `io.ekbatan.*` — Maven groupId and Java package don't need to match, and we kept the cleaner Java naming.
- **2 SMT shadow jars to GitHub Releases** as drop-in assets for Kafka Connect's `plugin.path`.

JReleaser orchestrates both from one config block in the root `build.gradle.kts`.

---

## Phase 0 — one-time setup

Do these once, before the first release. They never need to be repeated.

### 1. Sonatype Central Portal account

Sign up at <https://central.sonatype.com>. The Central Portal is the post-2024 publishing path; do not use the legacy OSSRH.

### 2. Claim the `io.github.zyraz-io` namespace

Sonatype's namespace convention is reverse-DNS of a thing you control. For projects hosted under a GitHub org, `io.github.<org>` is verified by creating a one-off repo whose name matches the verification code Sonatype generates.

In the Portal UI:
1. **Namespaces** → **Add Namespace** → enter `io.github.zyraz-io`.
2. Sonatype shows a verification code (looks like a short alphanumeric string, e.g. `abc123def456`).
3. Create a **public, empty** repo at `https://github.com/zyraz-io/<verification-code>` (use the exact code as the repo name).
4. Back in the Portal, click **Verify Namespace**. Sonatype checks the repo exists; verification is usually instant.

After verification you can delete the verification repo (or leave it — either is fine). The namespace is permanent.

### 3. Generate a GPG key pair

```bash
gpg --full-generate-key
# Choose: RSA and RSA, 4096 bits, no expiry (or 5y), name + email
# Set a strong passphrase — store it in your password manager. It CANNOT be recovered if lost.
```

Note the key ID (long form):

```bash
gpg --list-secret-keys --keyid-format LONG
# Look for: sec   rsa4096/<KEY_ID>
```

**Back up your keyring.** Without `~/.gnupg/` and the passphrase you can't sign future releases with this key — though you CAN rotate to a new key without much ceremony (see "Recovering from a lost GPG keyring" below). At minimum keep the auto-generated revocation certificate somewhere safe:

```bash
# Auto-generated by gpg when the key was created — keep this file in your password manager
# or on encrypted external storage. ~600 bytes, lets you revoke the key even after losing
# the rest of the keyring.
ls ~/.gnupg/openpgp-revocs.d/<KEY_ID>.rev
```

A full keyring backup is even better:

```bash
tar czf ~/gnupg-backup-$(date +%Y%m%d).tar.gz -C ~ .gnupg
# Move the .tar.gz to encrypted storage (NOT plain cloud sync, NOT the repo).
```

### 4. Publish the public half to a keyserver

Maven Central validates signatures against public keyservers. Publish to at least one of:

```bash
gpg --keyserver keys.openpgp.org --send-keys <KEY_ID>
gpg --keyserver keyserver.ubuntu.com --send-keys <KEY_ID>
```

`keys.openpgp.org` is the most widely-used one. Other servers tend to cross-sync.

### 5. Export the key pair for CI

JReleaser needs both halves as env vars. Export them in ASCII-armored form:

```bash
gpg --armor --export <KEY_ID> > public-key.asc
gpg --armor --export-secret-keys <KEY_ID> > secret-key.asc
```

You'll paste the contents of these files into GitHub repo secrets in the next step. After that, **delete** the `.asc` files from your machine — they're sensitive.

### 6. Generate a Central Portal User Token

In the Portal UI: **View Account** → **Generate User Token**. The dialog asks for:

- **Token Name**: pick something descriptive like `ekbatan-github-actions` (appears in audit logs)
- **Expiration Date**: `Does not expire` is fine for small OSS projects (the token is access-controlled by GitHub repo secrets anyway). Pick 6 months/1 year for rotation hygiene if you prefer.

You get a username/password pair separate from your Portal login — this is what the Publisher API uses. Save both halves now; **the password is shown only once**.

### 7. Store five secrets on the GitHub repository

Go to <https://github.com/zyraz-io/ekbatan/settings/secrets/actions> and add:

| Secret name | Value |
|---|---|
| `MAVENCENTRAL_USERNAME` | The User Token's username (from step 6) |
| `MAVENCENTRAL_PASSWORD` | The User Token's password (from step 6) |
| `GPG_PUBLIC_KEY` | Full contents of `public-key.asc` (including `-----BEGIN ... -----` and `-----END ... -----` lines) |
| `GPG_PRIVATE_KEY` | Full contents of `secret-key.asc` |
| `GPG_PASSPHRASE` | The passphrase you used when generating the key |

`GITHUB_TOKEN` is auto-provided by GitHub Actions — you don't add it manually.

That's it for Phase 0. Done once, reused for every release forever.

## Optional: client-side tag-validation hook

The release workflow already verifies that the pushed tag matches `gradle.properties` (see the "Verify gradle.properties version matches tag" step in `release.yml`). The repo also ships a **pre-push hook** at `.githooks/pre-push` that runs the same check **before the tag leaves your machine** — catches the mistake one CI run earlier.

To enable, run once per clone:

```bash
git config core.hooksPath .githooks
```

That points git at the version-controlled hooks directory (instead of the default `.git/hooks/`, which isn't tracked). From then on, any `git push` that includes a tag matching `v<X.Y.Z>` is checked against `gradle.properties` at the tagged commit — mismatches abort the push.

To bypass (rarely needed, e.g. pushing a non-release tag):

```bash
git push --no-verify
```

The hook is purely a local convenience — even if you skip enabling it, the workflow's guard step still catches the same mistake. Skipping it just means you find out from a failing CI run instead of from your terminal.

## Phase 1 — per-release workflow

Once Phase 0 is done, releasing a version is short. Examples below use `0.X.Y` as the next version number — replace with whatever you're shipping.

### Bump the version

Edit `gradle.properties`:

```properties
version=0.X.Y
```

Use bare `X.Y.Z` for a real release; `X.Y.Z-SNAPSHOT` for snapshot iteration on Central Portal's snapshot endpoint.

Commit:

```bash
git add gradle.properties
git commit -m "Release 0.X.Y"
git push origin main
```

### Tag and push the tag

```bash
git tag v0.X.Y
git push origin v0.X.Y
```

The tag push fires `.github/workflows/release.yml`, which runs:

```
./gradlew clean build
./gradlew publish              # → build/staging-deploy/ populated with 15 jars+sources+javadoc+POM
./gradlew jreleaserFullRelease # → signs, uploads to Sonatype staging
                               #   stops (does NOT auto-publish — see stage = UPLOAD config)
                               #   creates the GitHub Release with the SMT shadow jars
```

### Verify on the Central Portal

Log into <https://central.sonatype.com>. Under **Deployments** you'll see a new entry for `io.github.zyraz-io:0.X.Y`. Status will be **Validating** for ~1-2 min, then **Validated**.

Click into it. Check:

- All 15 modules listed
- Each has main jar + sources jar + javadoc jar + POM + signatures
- POM contents look right (group, artifact, version, license, developers, scm)

### Decide: Publish or Drop

- **Publish**: deployment goes live on Maven Central within ~10 min, syndicated to repo1.maven.org within ~30 min. Permanent. Cannot be deleted or overwritten.
- **Drop**: deployment is discarded. Version number is NOT burned — you can re-stage `0.X.Y` again with corrections.

This is the stage-and-confirm safety net. Use it.

### GitHub Release

Independent of the Central Portal flow, the release workflow has already created the GitHub Release at `https://github.com/zyraz-io/ekbatan/releases/tag/v0.X.Y` with the two SMT shadow jars attached:

- `ekbatan-debezium-smt-avro-0.X.Y.jar`
- `ekbatan-debezium-smt-protobuf-0.X.Y.jar`

GitHub Releases are editable and deletable — you can edit the release notes or remove a bad release without ceremony.

## Local dry runs

Before pushing a tag, validate locally:

```bash
# Verifies everything builds + publishes to ~/.m2/repository so you can
# point a sample app at the artifacts and check the POMs render correctly.
# No credentials needed for this one.
./gradlew clean build publishToMavenLocal
```

**JReleaser tasks need credentials even for config-validation.** `jreleaserConfig` doesn't make network calls but DOES enforce that all required env vars are set (GPG keys, Sonatype token, GitHub token). If you want to validate the JReleaser DSL locally, export the same env vars first:

```bash
export JRELEASER_GPG_PUBLIC_KEY="$(cat public-key.asc)"
export JRELEASER_GPG_SECRET_KEY="$(cat secret-key.asc)"
export JRELEASER_GPG_PASSPHRASE="<passphrase>"
export JRELEASER_GITHUB_TOKEN="<personal-access-token-with-repo-scope>"
export JRELEASER_MAVENCENTRAL_USERNAME="<user-token-username>"
export JRELEASER_MAVENCENTRAL_PASSWORD="<user-token-password>"

# Parses + reports the resolved JReleaser config; no network calls.
./gradlew jreleaserConfig

# Full release from your machine (skips GitHub Actions entirely):
./gradlew clean build publish jreleaserFullRelease
```

For a one-shot test that exercises every step but uploads nothing, JReleaser supports `--dry-run`:

```bash
./gradlew jreleaserFullRelease --dry-run
```

## What's recoverable, what isn't

| Step | Recoverable? |
|---|---|
| `SNAPSHOT` versions on Central Portal's snapshot endpoint | yes — each push overwrites |
| Tag pushed to git | yes — `git push --delete origin v0.0.1` removes |
| Release staged on Central Portal, not yet Published | yes — click "Drop" in the Portal UI |
| Release **Published** to Maven Central | **no — permanent** |
| GitHub Release (the SMT shadow jars) | yes — edit or delete via GitHub UI |

The only irreversible step is clicking **Publish** on the Central Portal. Until then, every other step can be redone.

## Troubleshooting

### The release workflow failed

1. Open the failed run: <https://github.com/zyraz-io/ekbatan/actions/workflows/release.yml>
2. The **"Upload JReleaser logs"** step runs `if: always()` and uploads `build/jreleaser/trace.log` + `build/jreleaser/output.properties` as a workflow artifact. Download these from the run page (top-right of the run UI, scroll to "Artifacts").
3. `trace.log` is verbose; search for `[ERROR]` to find the actual failure.

Common failures and fixes:

| Symptom | Cause | Fix |
|---|---|---|
| "Tag/version mismatch" | gradle.properties version doesn't match `v<X.Y.Z>` tag | Retag the bump commit, or fix gradle.properties and re-tag |
| "Signing key ... was NOT found as published" | Public key not on keys.openpgp.org yet (or wrong key) | Re-run `gpg --keyserver keys.openpgp.org --send-keys <KEY_ID>`; click the email confirmation link; wait a few minutes for propagation |
| "POMs failed to validate" | Missing scm/license/developer block, or coordinate mismatch | Inspect a POM in `build/staging-deploy/...` and compare to what Maven Central requires (see the validation list in the deployment view) |
| "Bundle upload failed: 401" | Wrong / expired Sonatype User Token | Generate a new token in the Portal, update `MAVENCENTRAL_USERNAME` / `MAVENCENTRAL_PASSWORD` GitHub secrets |
| "GPG: invalid passphrase" | `GPG_PASSPHRASE` secret doesn't match the key | Update the GitHub secret with the correct passphrase |

### Sonatype rejected the deployment

If the deployment reaches Sonatype but the Portal shows **Failed** status under Deployments:

1. Click the deployment in the Portal — it shows a per-file validation report
2. Common Sonatype rejections: missing `-sources.jar`, missing `-javadoc.jar`, missing `<scm>` in POM, missing `<licenses>` in POM, missing `<developers>` in POM, unsigned artifact
3. None of these should hit for us — the publishing convention plugin already covers all of them — but if a future change breaks one, the per-file report points right at the broken module
4. **Drop the failed deployment** in the Portal (it's non-published, so safe to drop), fix the issue, re-tag and re-push

### Maven Central syndication takes too long

The Central Portal publishes within ~10 min. The public `repo1.maven.org` Maven index typically syncs within 30 min. `search.maven.org` can take a few hours.

- Check the deployment status in the Portal — if it's **Published**, the artifact IS on Maven Central, just not yet syndicated everywhere
- Test direct fetch:
  ```bash
  curl -I https://repo1.maven.org/maven2/io/github/zyraz-io/ekbatan-core/<version>/ekbatan-core-<version>.jar
  # 200 = available; 404 = not syndicated yet, wait
  ```

## Recovering from a lost GPG keyring

If you lose `~/.gnupg/` (disk failure, lost laptop, etc.) you can rotate to a new key without major disruption. Old releases stay valid forever — they're signed with the old key, whose public half remains on `keys.openpgp.org`. New releases just use a new key.

```bash
# 1. Generate a new key (same flow as Phase 0 §3)
gpg --full-generate-key

# 2. Note the new KEY_ID
gpg --list-secret-keys --keyid-format LONG

# 3. Publish the new public key
gpg --keyserver keys.openpgp.org --send-keys <NEW_KEY_ID>

# 4. Export both halves
gpg --armor --export <NEW_KEY_ID> > public-key.asc
gpg --armor --export-secret-keys <NEW_KEY_ID> > secret-key.asc

# 5. Update the 3 GPG secrets in GitHub:
#      GPG_PUBLIC_KEY    → contents of public-key.asc
#      GPG_PRIVATE_KEY   → contents of secret-key.asc
#      GPG_PASSPHRASE    → new passphrase

# 6. Delete the local .asc files (treat secret-key.asc like a password)
rm public-key.asc secret-key.asc
```

Next release uses the new key automatically. Sonatype verifies each release's signature against whatever key matches the signature's ID — it doesn't care that previous releases used a different key.

If you saved the **revocation certificate** before losing the keyring (see Phase 0 §3), revoke the old key cleanly:

```bash
gpg --import <KEY_ID>.rev
gpg --keyserver keys.openpgp.org --send-keys <OLD_KEY_ID>
```

If you didn't save the revocation cert, the old key technically remains "valid" on keyservers — but inert, since you can't sign anything with it. For the typical "lost laptop, key wasn't exfiltrated" case, this is fine; the security risk is only if the private key was actually stolen.

## Future: auto-publish

Once you've done a few stage-and-confirm releases and trust the pipeline, flip the JReleaser config so releases auto-publish without a human click in the Sonatype Portal:

```kotlin
// In build.gradle.kts, jreleaser.deploy.maven.mavenCentral.sonatype:
//   change stage from UPLOAD to FULL (or PUBLISH — same effect for non-snapshot releases)
stage.set(org.jreleaser.model.api.deploy.maven.MavenCentralMavenDeployer.Stage.FULL)
```

With `FULL`, JReleaser uploads AND auto-clicks Publish on the deployment. Skip this until you've shipped a few releases manually.
